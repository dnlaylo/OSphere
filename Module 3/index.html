Module 3: Processes
Overview: This module focuses on the core concept of a process within an Operating Systems, defining it as a program in execution and explaining how the operating system manages active computing tasks. It covers the internal structure of a process, including its memory layout (text, data, stack, and program counter), and introduces the Process Control Block (PCB) as the key data structure used to store and manage process information. The module also explores process states and lifecycle transitions, scheduling queues and schedulers, context switching, and process creation and termination mechanisms such as those used in UNIX (fork, exec, wait, exit). Overall, it explains how operating systems coordinate multiple processes efficiently through scheduling, resource management, and state control to ensure smooth and organized system execution.
What is a Process? A process is a program in execution. It serves as the unit of work in most computer systems (such as time-sharing systems). Every process has a dedicated set of associated system resources and its execution must always progress in a sequential fashion.
The Program vs. Process Distinction A program by itself is a passive entity (such as a file stored on a disk), whereas a process is an active entity. • Although two or more processes may be associated with the exact same program, they are strictly considered separate execution sequences by the Operating System.
• Examples of this distinction: Several distinct users running different copies of an email program simultaneously, or a single user invoking a text editor several times. Each instance is a unique process.
Standard Examples of Processes: • Compiler
• Word-processor
• Email program
• Sending output to a printer
What a Process Includes? A process is structurally composed of the following four components:
• Text Section: Contains the executable program code.
• Data Section: Contains global and static variables.
• Stack: Contains temporary data generated during execution (e.g., function/method parameters, return addresses, and local variables).
• Program Counter: A specialized marker that specifies the exact next instruction to be executed by the process.
The Process Control Block (PCB) The Process Control Block (PCB) is a vital data structure in kernel memory that serves as the official representation of a process within an operating system. It acts as a comprehensive repository, tracking vital information like CPU registers, process state, memory boundaries, and open resources throughout the process's lifecycle. By saving this execution snapshot, the PCB enables core OS functions like multitasking and context switching, allowing the system scheduler to safely pause and resume processes without data loss.
Information Fields inside a PCB
Every process is managed via its PCB, which specifically tracks:
1. Process State: The current operational status of the process.
2. Program Counter: The memory address of the next instruction to be executed for this process. (This information must be saved when an interrupt occurs to allow the process to be successfully continued afterward).
3. CPU Registers: Hardware components including accumulators, index registers, stack pointers, and general-purpose registers. (This register data must also be saved during an interrupt to allow seamless resumption later).
4. CPU Scheduling Information: Includes process priority metrics, pointers to various scheduling queues, and other relevant scheduling parameters.
5. Memory-Management Information: Includes the direct values of the base and limit registers, page tables, or segment tables.
6. Accounting Information: Tracks the amount of CPU time and real time used, time limits, account numbers, and job or process numbers.
7. I/O Status Information: Includes a running list of I/O devices allocated to this specific process, a list of its open files, and related configurations.
Core Process States As a process executes, it continuously changes its operational state:
• New: The process is currently being created.
• Ready: The process is waiting to be assigned to a processor.
• Running: Instructions belonging to the process are actively being executed.
• Waiting (Blocked): The process is waiting for some specific event to occur (such as an I/O completion or receiving a signal).
• Terminated: The process has completely finished its execution.
Process Creation Events There are 4 principal events that trigger the creation of a process:
a. System initialization.
Execution of a process creation system call by an actively running process.
b. A direct user request to create a new process.
c. Initiation of a background batch job.
• State Entry: On creation, the process immediately enters the New state, and its initial control block (e.g., PCB1) is allocated.
Process Termination Events A process transitions to the Terminated state due to one of 4 specific causes:
a. Normal exit (Voluntary).
b. Error exit (Voluntary).
c. Fatal error (Involuntary).
d. Killed by another process (Involuntary).
Process Scheduling Queues and Schedulers 1. The Three Scheduling Queues
• As processes move through their lifecycle, they are organized into targeted queues:
• Job Queue: The set consisting of all processes currently residing in the entire system.
• Ready Queue: The set of all processes residing in main memory that are completely ready and waiting to execute.
• Device Queues: The sets of processes strictly waiting for a specific, individual I/O device to become available. 2. Types of System Schedulers
• Long-Term Scheduler (or Job Scheduler): Selects which processes should be moved from the disk's input queue and brought into main memory's ready queue.
• Directly controls the degree of multiprogramming (the total number of processes residing in memory).
• Invoked very infrequently (intervals of seconds or minutes), meaning it can afford to be relatively slow.
• Must maintain a balanced Process Mix: It balances I/O-bound processes (which spend more time doing I/O than computations and have many short CPU bursts) and CPU-bound processes (which spend more time doing computations and have a few, very long CPU bursts).
• Short-Term Scheduler (or CPU Scheduler): Selects which process residing in the main memory's ready queue should be executed next and allocates the CPU to it.
• Invoked very frequently (intervals of milliseconds), meaning it must be extremely fast to maintain system performance.
• Medium-Term Scheduler: An intermediate-level scheduling system utilized in time-sharing or virtual memory environments.
• Removes processes entirely from main memory and saves them temporarily to disk (Swapping), reducing the degree of multiprogramming when memory is constrained. The processes are reintroduced later to resume exactly where they left off.

3. The Dispatcher
• Dispatcher: A specific module that gives physical control of the CPU to the exact process selected by the short-term scheduler.
Step-by-Step Execution Journey of a Process 1. Creation: An arriving job enters the system and is placed on the Job Queue in the New state.
2. Memory Admission: If there isn't enough room in main memory, the jobs wait on the disk in an Input Queue (a collection of processes on disk waiting to be brought into memory). The Long-Term Scheduler selects a job, loads it into main memory, puts it in the Ready Queue, and its state becomes Ready.
3. CPU Dispatching: The Short-Term Scheduler selects a process from the Ready Queue. The Dispatcher hands over the CPU, and the process enters the Running state, actively accessing instructions and data from memory.
4. Runtime Transitions: While running, one of four events will happen to the process:
• Event 1 (I/O Request): The process issues an I/O request. It is placed in a specific I/O Device Queue and its state shifts to Waiting. Once the I/O or event completion occurs, it switches back to the Ready Queue.
• Event 2 (Time Expiration): The maximum time slice allotted to the process in the CPU expires. It is intercepted, removed from the CPU, and placed back into the Ready Queue.
• Event 3 (Child Creation): The process creates a new subprocess (forks a child) and is forced to wait until that child finishes, remaining in the Waiting state.
• Event 4 (Interrupt): The process is forcibly removed from the CPU as a result of an external interrupt and is placed directly back into the Ready Queue.
5. Termination: Upon completing its final statement or encountering an error, the process finishes execution, its memory space is declared available, and its control block enters the Terminated state.
Advanced Concept: Memory Swapping and Suspended States
If the processor is much faster than the I/O devices, all available processes could eventually find themselves waiting for I/O simultaneously. To optimize memory, the OS swaps these blocked processes onto the disk to free up main memory for new jobs. This introduces two secondary-memory states:
• Blocked/Suspend: The process is currently located in secondary memory (disk) and is awaiting an external I/O event.
• Ready/Suspend: The process is located in secondary memory (disk) but is fully available for execution the moment it is swapped back into main memory.
Context Switching
1. Core Mechanics
• When the CPU switches focus from one process to another, the system must securely save the state of the old process and load the saved state for the new process.
• The saved state consists of the exact CPU registers, the current process state, and all memory-management information.

2. System Overhead
• Context-switch time is pure system overhead. The operating system and hardware do no useful work for the user while a switch is taking place.
• Context-switch performance and speed are highly dependent on direct hardware support.
Operations on Processes
1. Process Creation and Trees
• Processes are organized hierarchically. A parent process can create children processes, which, in turn, can create other sub-processes, forming a structural tree of processes.
• Resource Sharing Options: • The parent and children share all system resources.
• The children share a strict subset of the parent’s resources.
• The parent and child share no resources at all.
• Execution Behavior:
• The parent and children execute concurrently.
• The parent blocks and waits until some or all of its children have completely terminated.
• Address Space Behavior:
• The child process is an exact structural duplicate of the parent process.
• The child process has an entirely new program loaded into its address space.
2. Concrete UNIX Implementations
• fork(): A system call that creates a brand-new process. The newly created child process is an initial duplicate of the parent process.
• exec(): A system call typically used immediately after a fork() to replace the process’s memory space with an entirely new program layout.
• wait(): A system call through which a parent process receives output data from its child and suspends itself until the child finishes.
• exit(): A system call used by a process to execute its last statement and ask the operating system to delete it.
3. Cascading Termination
• A strict operating system policy that does not allow a child process to continue running if its parent process terminates. If a parent terminates (either normally or via an abort), the operating system automatically and recursively terminates all of its children.
