Module 4: CPU Scheduling
Overview: This module explains the fundamental principles behind how an Operating Systems manages process execution and CPU allocation. It introduces the core components of CPU scheduling, including the ready queue, short-term scheduler, dispatcher, and dispatch latency, as well as the CPU–I/O burst cycle that describes how processes alternate between computation and waiting. The module then explores scheduling decisions and performance criteria such as CPU utilization, throughput, turnaround time, waiting time, and response time, which guide scheduling efficiency. It also examines major CPU scheduling algorithms—FCFS, SJF, Priority Scheduling, Round Robin, HRRN, Multilevel Queue, and Multilevel Feedback Queue—highlighting their behavior, advantages, and trade-offs. Overall, the module provides a structured understanding of how operating systems optimize process execution to ensure fairness, efficiency, and system responsiveness.
The Core Architecture The Ready Queue : As a process is loaded into the main memory, it is kept in the ready queue. The ready queue contains the set of all processes residing in main memory that are ready and waiting to execute. Note: The ready queue is not necessarily a First-In, First-Out (FIFO) queue.

Short-term Scheduler (CPU Scheduler) : A system module that selects from among the processes in main memory that are ready to execute, and allocates the CPU to one of them.

Dispatcher : A specific module that gives control of the CPU to the process selected by the short-term scheduler.

Dispatch Latency : The total time it takes for the dispatcher to stop one process from running and start another one executing.

Objective of Multiprogramming: To achieve maximum CPU utilization by always having some process running.

CPU Scheduling : The fundamental basis of multiprogrammed operating systems. It deals with the problem of deciding which of the processes in the ready queue is to be allocated the CPU.
When the Dispatcher gives control of the CPU to a selected process, it executes the following sequential steps:

Switch Context: Saves the state (context) of the currently running process and loads the saved state of the new process.
Switch to User Mode: Flips the CPU execution mode from kernel/privileged mode to user mode.
Jump to Location: Jumps to the proper location in the target user program to restart or begin execution of that program.

The CPU-I/O Burst Cycle Concept Definition: Process execution consists of an alternating cycle of CPU execution (computations) and I/O wait (input/output operations).

CPU Burst: The time period where the process is actively utilizing the CPU (e.g., executing instructions like load, store, add, increment index).

I/O Burst: The time period where the process is blocked, waiting for an input/output operation to complete (e.g., reading from a file, writing to a disk).

Histogram of CPU Burst Times: CPU burst durations are characteristically hyperexponential. There is a very high frequency of short CPU bursts and a very low frequency of long CPU bursts.

Classification of Processes:

I/O-Bound Process: Spends more of its time doing I/O operations than computations; characterized by many short CPU bursts.

CPU-Bound Process: Spends more of its time doing computations; characterized by a few, very long CPU bursts.
 Scheduling Decisions & Schemes CPU scheduling decisions take place under four specific process state changes:
When a process switches from the running state to the waiting state (e.g., an I/O request).
When a process switches from the running state to the ready state (e.g., a timer interrupt).
When a process switches from the waiting state to the ready state (e.g., I/O completion).
When a process terminates.

Non-preemptive Scheduling (Cooperative): Scheduling that occurs only under conditions 1 and 4. There is no alternative choice; once a process is allocated the CPU, it continues running until it voluntarily terminates or blocks itself to wait for I/O.

Preemptive Scheduling: Scheduling that occurs under conditions 2 and 3. The operating system has a choice and can interrupt a currently running process to move it back to the ready state, preventing any single process from monopolizing the processor for too long.
Scheduling Performance Criteria CPU Utilization: The percentage or fraction of time that the CPU is actively doing useful work.
Target: Maximize CPU utilization.

Throughput: The absolute number of processes that are fully completed per unit of time.
Target: Maximize throughput.

Turnaround Time: The entire time interval spanning from the exact moment a process is submitted to the moment of its complete completion.
Mathematical Formula: Turnaround Time = Finish Time - Arrival Time
Target: Minimize turnaround time.

Waiting Time: The cumulative amount of time a process spends sitting and waiting inside the ready queue.
Mathematical Formula: Waiting Time = Turnaround Time - Burst Time
Target: Minimize waiting time.

Response Time: The specific amount of time it takes from process submission until the very first response or output begins to be produced (not the time it takes to output the entire response).
Target: Minimize response time.
CPU Scheduling Algorithms First-Come, First-Served (FCFS) Scheduling Definition: The core algorithm where the process that requests the CPU first is allocated the CPU first, based entirely on its arrival time or when it was loaded into memory.

Data Structure: Managed using a standard First-In, First-Out (FIFO) queue.

Scheme Type: Strictly Non-preemptive. Once a process gets CPU allocation, it retains it until it terminates or requests an I/O operation.

Key Characteristics & Limitations:
Average waiting time is generally not minimal.
A short process may be forced to wait a very long time if a massive process arrives ahead of it.
It inherently favors CPU-bound processes over I/O-bound processes.
The Convoy Effect: A major performance phenomenon where multiple short processes end up stuck waiting in a long line behind one massive, slow process while it holds the CPU.
Shortest-Job-First (SJF) / Shortest-Job-Next (SJN) Definition: An algorithm that allocates the CPU based on the length of the process's next CPU burst (more accurately called "Shortest Next CPU Burst"). Short processes are allowed to jump directly ahead of longer processes.

Tie-Breaker Rule: If two processes share the exact same next CPU burst length, FCFS scheduling is employed to break the tie.

Key Performance Characteristics:
Advantage: It is mathematically optimal—it guarantees the minimum possible average waiting time for any given set of processes.

Disadvantage: Can lead to Starvation, where longer processes may wait indefinitely if a continuous stream of shorter processes keeps arriving in the ready queue.

Operational Schemes (Two Variants):
Non-preemptive SJF: Once the CPU is allocated to a process, it cannot be interrupted or overridden until it completely finishes its current CPU burst, even if a new process with a shorter burst arrives.

Preemptive SJF / Shortest-Remaining-Time-First (SRTF): If a new process arrives in the ready queue with a next CPU burst time that is shorter than the remaining burst time of the currently executing process, the running process is immediately preempted and swapped out.
Priority Scheduling Definition: An algorithm where a priority value (typically an integer) is explicitly associated with every process, and the CPU is allocated to the process possessing the highest priority.

Standard Rule: The smallest integer value represents the highest priority level.

Relationship to SJF: SJF is simply a specific form of priority scheduling where the priority value is equal to the predicted length of the next CPU burst.

Key Issues & Solutions:
Problem - Starvation (Indefinite Blocking): Low-priority processes may sit in the queue forever and never execute if higher-priority processes are constantly submitted.
Solution - Aging: A technical process where the operating system gradually increases the priority level of long-waiting processes as time progresses.

Operational Schemes (Two Variants):
Nonpreemptive Priority (NPP): The CPU is given to the job with the highest priority. If priorities are equal, FCFS is used. The process cannot be interrupted while running.

Preemptive Priority (PP): If a newly arrived process features a higher priority level than the process currently executing, the running job is instantly preempted.
Definition: An algorithm designed specifically for time-sharing environments. It operates similarly to FCFS, but overlays preemption mechanisms to dynamically cycle through executing processes.

Time Quantum (q): A small, fixed unit of CPU time (typically ranging between 10 to 100 milliseconds) allocated to a process at a time. No single job can continuously hold the CPU for longer than one time quantum per turn.

Preemption Mechanism: Driven by clock interrupts generated at periodic intervals matching the quantum size. When an interrupt occurs, the running process is preempted and placed at the back of the ready queue.

Wait-Time Guarantee: If there are n processes in the ready queue and the time quantum is q, each process receives 1/n of the CPU time in chunks of at most q. No process will ever wait more than (n-1) x q time units for its next turn.

Performance Dependency on Quantum Size:
If q is extremely large, RR performance degenerates and becomes identical to FIFO/FCFS.

If q is extremely small, q must still remain significantly larger than the system's context switch time. Otherwise, the performance overhead from constant switching becomes unsustainably high.

Turnaround Optimization Rule:* Average turnaround time improves if the majority of processes can fully complete their next CPU burst within the span of a single time quantum. Round Robin Scheduling
Definition: A non-preemptive scheduling algorithm where the process with the highest mathematically computed response ratio is chosen for execution next.

Response Ratio Formula:
Response Ratio = (Time Spent Waiting + Expected Service TIme) / Expected Service Time
(Where "Expected Service Time" is equal to the next CPU burst time)

Tie-Breaker Rule: If two processes compute down to an identical response ratio, FCFS is utilized.

Key Characteristics:
The absolute minimum possible value of a response ratio is 1 (ocruns the exact moment a process enters the ready queue).

Shorter processes are naturally favored (since a smaller denominator yields a higher ratio).

Longer processes are protected against starvation via aging—as their waiting time (numerator) continuously grows over time, their overall response ratio increases until they win CPU allocation.
Highest-Response-Ratio-Next (HRRN) Scheduling
Definition: A scheduling methodology where the ready queue is partitioned into several entirely separate sub-queues based on process characteristics (e.g., separating foreground interactive processes from background batch processes).

Independent Algorithms: Every sub-queue maintains its own separate scheduling algorithm (for example, the foreground queue might run Round Robin while the background queue runs FCFS).

Inter-Queue Scheduling Methods (Managing Queue Priority):

Fixed Priority Preemptive Scheduling: Each individual queue holds absolute priority over lower-priority queues. No process in a lower-priority queue (e.g., a batch queue) can run unless all higher-priority queues are completely empty. If a lower-priority process is currently running and a process enters a higher-priority queue, the lower-priority process is immediately preempted.
Time Slicing: Every queue receives a statically allocated, dedicated percentage of the total CPU time to divide among its own processes (e.g., Queue 1 takes 50% of CPU time, Queue 2 takes 30%, and Queue 3 gets 20%).
Multilevel Queue Scheduling
Multilevel Feedback Queue (MLFQ) Scheduling Definition: An advanced extension of Multilevel Queues where processes are not permanently assigned to a single queue. Instead, processes are permitted to dynamically move between different queues based on their execution history. This structure inherently implements aging and penalizes processes that monopolize the CPU for long periods.

System Parameters (Defining an MLFQ System) : An MLFQ scheduler is strictly defined by five distinct parameters:

The total number of sub-queues.
The specific scheduling algorithm used within each individual queue.
The explicit method used to determine when to upgrade a process to a higher-priority queue (Aging).
The explicit method used to determine when to demote a process to a lower-priority queue.
The explicit method used to determine which queue a process will initially enter when it requires service.

The Demotion/Interruption Process (Step-by-Step Multi-Level Life Cycle):

A brand new process enters the highest-priority queue (Q0) under an FCFS structure and is allocated a small time quantum.
If the process finishes its job within that time quantum, it terminates cleanly.
If the quantum expires and the process has not finished, it is preempted, demoted, and moved to the back of the next lower-priority queue (Q1).
In Q1, the process waits until Q0 is completely empty. When it executes, it receives an expanded time quantum.
If it still fails to complete within the Q1 quantum, it is demoted down again to a lower queue (Q2), which often operates under a basic FCFS structure.
Preemption Rule: Lower-priority queues are processed only when all higher-priority queues are completely empty. If a process enters a higher queue, execution in the lower queue is stopped.

Key Issues & Solutions:

Problem: Starvation of long processes if I/O-bound and interactive processes keep arriving continuously.
Solution: Aging (processes waiting too long in a low-priority queue can be dynamically moved to a higher-priority queue).
