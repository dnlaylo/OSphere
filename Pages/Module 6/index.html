Virtual Memory Module 6:
Overview:
This module explains how an Operating Systems uses virtual memory to overcome the limitations of physical RAM and allow large programs to execute efficiently even when they are only partially loaded into memory. It introduces the concept of virtual memory as an abstraction that separates logical and physical address spaces, enabling partial process loading, memory sharing, and improved system utilization. The module further explores demand paging as the primary implementation technique, where pages are loaded only when referenced, and discusses key mechanisms such as page tables, valid/invalid bits, and page faults. It also covers performance considerations using effective access time and examines page replacement strategies like FIFO, Optimal, LRU, LFU, and Second Chance, along with frame allocation policies that determine how memory is distributed among processes. Overall, the module highlights how modern operating systems efficiently manage limited memory resources to support multitasking and high performance.
Memory Management and the Need for Virtual Memory
Memory management is a fundamental function of the operating system that controls how memory resources are allocated, used, and protected among multiple processes. Modern operating systems support multiprogramming, where several processes execute concurrently and compete for limited physical memory resources.
As programs become larger and more complex, it becomes impractical to require every process to be loaded entirely into main memory before execution. Many portions of a program may not be needed immediately or may never be used during a particular execution. Examples include error-handling routines, rarely used program features, dynamically allocated data structures, and optional modules.
To overcome the limitations imposed by physical memory size, operating systems implement virtual memory, which allows processes to execute even when only part of their address space is resident in main memory.
Virtual Memory
Virtual memory is a memory management technique that creates the illusion of a large, contiguous address space for each process, independent of the actual size of physical memory.
It separates a process's logical address space from the system's physical address space. Memory references generated by the CPU use logical addresses, which are translated into physical addresses through hardware and operating-system mechanisms.
Only the portions of a process that are actively needed are maintained in physical memory, while the remaining portions reside on secondary storage until they are required.
Characteristics of Virtual Memory
Virtual memory possesses several important characteristics:
Partial Process Loading
Only the required portions of a program need to be loaded into memory for execution. Unused pages remain on secondary storage until referenced.
Large Logical Address Space
The logical address space available to a process can be significantly larger than the available physical memory.
Memory Sharing
Multiple processes may share portions of their address spaces, such as shared libraries, without duplicating memory contents.
Process Isolation and Protection
Address translation mechanisms provide protection by preventing processes from directly accessing each other's memory spaces.
Efficient Memory Utilization
Physical memory is allocated only to actively used portions of processes, reducing waste and improving system efficiency.
Benefits
• Enables execution of programs larger than physical memory.
• Increases the degree of multiprogramming.
• Improves overall CPU utilization.
• Reduces memory waste by loading only required pages.
• Supports memory sharing among processes.
• Simplifies process creation through mechanisms such as copy-on-write and shared pages.
Virtual Memory that is larger than Physical Memory
Virtual Memory Implementation:
• Demand Paging
Memory is divided into pages, and pages are loaded into memory only when they are referenced.
• Demand Segmentation
Segments are brought into memory only when required by a process.
• Paged Segmentation
Combines the advantages of segmentation and paging by dividing segments into pages. The primary implementation discussed in this module is Demand Paging.
Lazy Pager
Demand paging employs a lazy pager, sometimes called a lazy swapper.
The lazy pager delays transferring pages from secondary storage to main memory until the page is explicitly referenced. As a result, pages that are never accessed are never loaded.
Page Table Support
Demand paging relies on page tables that maintain information about each page.
Each page-table entry contains status information indicating whether a page currently resides in physical memory.
Steps in Demand Paging
1. Page Selection: When a process is to be swapped into memory, the pager predicts which pages are likely to be used before the process is swapped out again.
2. Loading Required Pages: The pager loads only the necessary pages into physical memory, reducing swap time and minimizing the amount of physical memory required for process execution.
By loading only the pages that are likely to be used, demand paging improves memory utilization and reduces the overhead associated with loading an entire process into memory.
Valid Bit: A valid bit indicates that:
1. The page belongs to the process's logical address space.
2. The page is currently resident in physical memory.
Invalid Bit: An invalid bit indicates either:
1. The page is not currently resident in memory, or
2. The referenced address is not part of the process's legal address space.
3. An access to an invalid page entry triggers a page-fault exception.
Page Faults
A page fault is an exception generated when a process references a page that is not currently resident in physical memory.
Page faults are expected events in demand-paging systems and are handled by the operating system.
What Happens When a Referenced Page is Not in Memory?
When a process attempts to access a page that is not currently resident in physical memory, a page fault occurs. The hardware detects that the page-table entry is marked invalid for the current memory access and generates a page-fault exception. Control is then transferred to the operating system, which determines whether the reference is valid and retrieves the required page if necessary. Control is then transferred to the operating system, which determines whether the memory reference is valid and takes the necessary actions to retrieve the required page from secondary storage.
Page Fault Handling Procedure
When a page fault occurs:
• Hardware detects the invalid page-table entry.
• Control transfers to the operating system.
• The operating system verifies that the memory reference is valid.
• The required page is located on backing storage.
• A free frame is obtained, or a victim page is selected if no free frame exists.
• The required page is loaded into memory.
• The page table is updated.
• The interrupted instruction is restarted.
Pure Demand Paging
Pure demand paging refers to a system in which no pages are loaded into memory before execution begins.
Every page enters memory only after its first reference, causing an initial page fault for each page that must be accessed.
Performance of Demand Paging
The efficiency of a demand-paging system depends largely on the page-fault rate. Memory accesses typically require nanoseconds or microseconds, whereas servicing a page fault may require milliseconds because secondary storage operations are involved.
Consequently, even a relatively small page-fault rate can significantly impact system performance.

Page Replacement
Need for Page Replacement
Page replacement is required when a page fault occurs and no free memory frames are available. The operating system must remove an existing page from memory to create space for the required page.
Options When No Free Frames Are Available
When a page fault occurs and no free memory frames remain available, the operating system must choose one of the following actions:
1. Terminate the Process
• Stop the process that caused the page fault.
• This option is rarely used because it defeats the purpose of demand paging, which aims to improve CPU utilization and throughput.
2. Swap Out a Process
• Temporarily move an entire process to secondary storage.
• This frees all frames used by the process but reduces the degree of multiprogramming.
3. Perform Page Replacement
• Select a victim page currently residing in memory.
• Remove the victim page and load the required page into the freed frame.
• This is the most common approach in modern operating systems.
How Page Replacement Works
1. Locate the required page in secondary storage (backing store).
2. Check for a free frame.
• If a free frame exists, load the page into it.
• If no free frame exists, use a page replacement algorithm to select a victim page.
3. If the victim page has been modified, write it back to disk and update the page table.
4. Load the required page into the freed frame and update the page table.
5. Restart the interrupted process.
Victim Page
The page selected for removal is called the victim page. If it has been modified since being loaded into memory, it must be written back to secondary storage before replacement.
Objectives of Page Replacement
A page replacement algorithm aims to:
• Minimize page faults
• Maximize memory utilization
• Reduce disk I/O operations
• Improve overall system performance
Implementation Requirements Demand paging requires:

• A page replacement algorithm to determine which page should be replaced.
• A frame allocation algorithm to determine how many memory frames are assigned to each process.

Page Replacement Algorithms (PRA)
Page Replacement Algorithms (PRAs) are techniques used by the operating system to determine which memory page should be selected as the victim page when a page fault occurs and no free memory frames are available.
The primary objective of a page replacement algorithm is to minimize the page-fault rate while maintaining efficient memory utilization and reducing disk input/output operations. Because page-fault handling involves accessing secondary storage, a lower page-fault rate generally results in better system performance.
Reference String Evaluation
The effectiveness of a page replacement algorithm is evaluated by analyzing its performance on a reference string, which is a sequence of page references generated during program execution.
Evaluation typically involves the following steps:
1. Select a reference string.
2. Assume a fixed number of available memory frames.
3. Apply the page replacement algorithm to the reference string.
4. Count the total number of page faults generated.
The algorithm that produces the lowest number of page faults for a given workload is generally considered more efficient.
First In First Out (FIFO)
This is the simplest page replacement algorithm. In this algorithm, the operating system keeps track of all pages in the memory in a queue, the oldest page is in the front of the queue. When a page needs to be replaced, the page in the front of the queue is selected for removal.
Example: Consider page reference string 1, 3, 0, 3, 5, 6, 3 with 3-page frames. Find the number of page faults using FIFO Page Replacement Algorithm.
• Initially, all frames are empty. When pages 1, 3, and 0 are referenced, they are loaded into the available frames, resulting in 3 page faults.
• When page 3 is referenced next, it is already present in memory, resulting in a hit and no page fault.
• When page 5 is referenced, it is not present in memory. Following the FIFO algorithm, the oldest page (1) is removed from the top frame, and the remaining pages shift up, allowing page 5 to be loaded into memory, resulting in 1 page fault.
• When page 6 is referenced, it is also not present in memory. The next oldest page (3) is removed from the top frame as the pages shift up, and page 6 is loaded into memory, resulting in 1 page fault.
• Finally, when page 3 is referenced again, it is no longer present in memory. Following FIFO, the oldest page currently in memory (0) is removed from the top frame, and page 3 is loaded at the bottom, resulting in 1 page fault.
Total Page Faults = 6
Optimal Page Replacement
In this algorithm, pages are replaced which would not be used for the longest duration of time in the future.
Example: Consider the page references 7, 0, 1, 2, 0, 3, 0, 4, 2, 3, 0, 3, 2, 3 with 4-page frame. Find number of page faults using Optimal Page Replacement Algorithm.
• Initially, all slots are empty, so when 7 0 1 2 are allocated to the empty slots ---> 4 Page faults
• The next reference, 0, is already present in memory and results in a hit. When page 3 is referenced, page 7 is selected for replacement because it will not be used again in the remaining reference string, resulting in 1 page fault.
• The next reference, 0, is already present in memory and results in a hit. When page 4 is referenced, page 1 is selected for replacement because it will not be used again in the remaining reference string, resulting in 1 page fault.
Total Page Faults = 6
All remaining page references are already present in memory and therefore result in hits.
Optimal page replacement is perfect, but not possible in practice as the operating system cannot know future requests. The use of Optimal Page replacement is to set up a benchmark so that other replacement algorithms can be analyzed against it.
Least Recently Used (LRU)
In this algorithm, a page will be replaced which is least recently used.
Example: Consider the page reference string 7, 0, 1, 2, 0, 3, 0, 4, 2, 3, 0, 3, 2, 3 with 4-page frames. Find number of page faults using LRU Page Replacement Algorithm.
• Initially, all four frames are empty. The first four page references (7, 0, 1, and 2) are loaded into the available frames, resulting in 4 page faults.
• The next reference, 0, is already present in memory, resulting in a hit.
• When page 3 is referenced, page 7 is the least recently used page and is therefore replaced, resulting in 1 page fault.
• The next reference, 0, is already present in memory and results in a hit.
• When page 4 is referenced, page 1 is the least recently used page and is replaced, resulting in 1 page fault.
• The remaining references (2, 3, 0, 3, 2, 3) are already present in memory and therefore result in hits.
Total Page Faults = 6
LRU Approximation (Second Chance / Clock)
In this algorithm, the operating system approximates the behavior of the Least Recently Used (LRU) algorithm by using a reference bit associated with each page. A page with a reference bit of 0 is selected for replacement. If a page has a reference bit of 1, it is given a second chance and its reference bit is cleared before the search continues.
Example: Consider the page reference string 7, 0, 1, 2, 0, 3, 0, 4, 2, 3, 0, 3, 2, 3 with 4-page frames.
• Initially, all four frames are empty. The first four page references (7, 0, 1, and 2) are loaded into memory, resulting in 4 page faults.
• The next reference, 0, is already present in memory, resulting in a hit.
• When page 3 is referenced, no free frame is available. The clock pointer begins scanning the pages. Pages whose reference bit is 1 are given a second chance and their reference bits are cleared to 0. The first page encountered with a reference bit of 0 is page 7, which is selected for replacement. This results in 1 page fault.
• The next reference, 0, is already present in memory and results in a hit.
• When page 4 is referenced, the clock pointer again scans the pages starting from its current position. Page 0 (reference bit = 1) is given a second chance and its bit is cleared to 0. The pointer moves forward, and the first page encountered with a reference bit of 0 is page 1, which is selected as the victim page. Therefore, page 1 is replaced by page 4, resulting in 1 page fault.
• The remaining references (2, 3, 0, 3, 2, 3) are already present in memory and therefore result in hits.
Total Page Faults = 6
Note: LRU Approximation uses less overhead than true LRU while providing similar performance in many workloads.
Counting-Based Page Replacement (LFU)
Counting-Based Page Replacement maintains a reference counter for each page. The page with the smallest reference count is selected for replacement. This approach is known as Least Frequently Used (LFU) page replacement.
Example: Consider the page reference string 7, 0, 1, 2, 0, 3, 0, 4, 2, 3, 0, 3, 2, 3 with 4-page frames.
• Initially, all four frames are empty. The first four page references (7, 0, 1, and 2) are loaded into memory, resulting in 4 page faults.
• The next reference, 0, is already present in memory. Its reference count is increased, resulting in a hit.
• When page 3 is referenced, all frames are occupied. Under the LFU policy, the page with the smallest reference count is selected for replacement. Since page 7 has the lowest reference count, it is replaced by page 3, resulting in 1 page fault.
• The next reference, 0, is already present in memory and therefore results in a hit.
• When page 4 is referenced, pages 1, 2, and 3 have the same lowest reference count. Because of this tie, FIFO (First-In, First-Out) is used as the tie-breaking rule. Among these pages, page 1 is the oldest and is therefore selected for replacement. Page 1 is replaced by page 4, resulting in 1 page fault.
• The remaining references (2, 3, 0, 3, 2, and 3) are already present in memory. Their reference counters are updated accordingly, and all of these references result in hits.
Total Page Faults = 6
Note: In this example, Optimal, LRU, and LRU Approximation (Second Chance) produce the same number of page faults. This does not mean that all page replacement algorithms always perform equally. The number of page faults depends on the reference string, the number of available frames, and the replacement strategy used. Different workloads may produce significantly different results for each algorithm.

Frame Allocation
Frame allocation is the process by which the operating system distributes available physical memory frames among active processes. Proper frame allocation is essential for efficient memory management and system performance in a virtual memory environment.
Characteristics of Frame Allocation
• The operating system cannot allocate more frames than the total number of frames available in physical memory.
• The number of frames assigned to a process directly affects its performance. Allocating fewer frames generally increases the page-fault rate.
• The minimum number of frames required by a process is determined by the system architecture and instruction requirements.
• The maximum number of frames that can be allocated depends on available physical memory and the needs of other processes.
Effective frame allocation helps balance memory usage among processes, minimizes page faults, and improves overall system performance. To support demand paging, the operating system must employ a frame allocation strategy that determines how available frames are distributed among competing processes.
