Module 5: Memory Management
Overview: This module explains how an Operating Systems handles the allocation, tracking, protection, and optimization of main memory to ensure efficient execution of processes. It introduces the step-by-step flow of how programs move from disk storage into RAM through schedulers, how memory is assigned, used, and freed during execution, and why memory management is essential for supporting multiprogramming and system performance. The module covers key concepts such as address binding, logical and physical address translation, the Memory Management Unit (MMU), dynamic loading and linking, swapping, and memory protection using base and limit registers. It also explores memory allocation strategies (MFT and MVT), fragmentation issues, paging systems, and address conversion, highlighting how operating systems balance flexibility, efficiency, and safety in managing physical memory resources.
Memory Management Memory management is the system mechanism that coordinates the loading, tracking, and freeing of physical RAM as active processes move between disk storage and the processor. Because the CPU can only execute code located inside Main Memory, the operating system uses specialized long-term and CPU schedulers to control the flow of data pipelines, keep memory safe from accidental overwrites, and maximize hardware efficiency.
Step-by-Step Paging Hardware Action
When a program runs, the CPU generates a logical address. The hardware converts it into a physical address through these precise steps:
1. Separate the Address: The hardware splits the logical address into two parts: a page number ($p$) and a page offset ($d$).
2. Look up the Page Table: The system uses the page number ($p$) as an index to find the matching entry in the page table.
3. Extract the Frame: From that page table entry, the hardware extracts the specific physical frame number ($f$) where that page is living.
4. Combine and Locate: The hardware combines the frame number ($f$) with the original page offset ($d$) to create the final physical address. The CPU can now read or write data directly from that frame spot.
Motivation of Memory Management
• To share CPU and implement CPU scheduling, several processes must be kept in memory at a certain time;
• To realize an increase in performance, memory has to be shared; affects the degree of multiprogramming;
There is a need to have algorithms to manage memory.
What is Memory? Memory is a large, rapidly accessible array of words or bytes, where each individual location is assigned its own unique physical address. It serves as the primary, volatile hardware workspace that sits directly between long-term disk storage and the processor, acting as the mandatory repository where a program's binary instructions and data must be loaded so that the CPU can directly fetch, decode, and execute them in real time.
Addresses
• Symbolic Addresses: These are the plain-text variable names, function identifiers, or object references you type out in your source code (for example: int total, float price, or count).
• Relocatable Addresses: When you compile your source code, the compiler translates those text names into relative offsets (for example: "14 bytes from the beginning of this module"). They are "relocatable" because the code can be placed anywhere in memory, and the offsets will still work.
• Absolute Addresses: This is the final hardware stage. When the program is actually loaded into RAM to run, a linkage editor or system loader binds those relative offsets to exact, physical spots on your memory chips (for example: the precise hardware address 0x7FFF001A).
Binding is the formal process of mapping a program's references from one memory address space to another. When you write code, your variable names cannot be understood directly by the computer hardware; binding translates those friendly source code names into temporary system locations, and eventually into exact, permanent hardware positions in your RAM so the CPU can run them.
Address binding
1. Compile Time (Deciding the shelf number while writing the book)
• The compiler figures out exactly where the program will sit in the RAM before the program even runs. It hardcodes those exact physical addresses into the machine code (called absolute code).
• You must know the exact starting location in memory ahead of time (a priori). If that spot in RAM is occupied later, or if you want to move the program to a different starting location, the program won't work. You have to go back, change the starting location, and recompile the whole thing.
2. Load Time (Deciding the shelf number when putting the book on the shelf)
• If the compiler doesn't know where the program will sit in RAM during compilation, it keeps things flexible. It generates relocatable code, using placeholder positions (like "offset 0, offset 1, offset 2").
• The actual, final memory addresses aren't assigned until the moment you double-click the program and the OS loads it into RAM. Once it is loaded, those placeholders turn into permanent physical positions for that specific run. If you want to move it while it's running, you can't.
3. Execution Time / Run Time (Changing the shelf number while reading the book)
• This is the most flexible approach and the one modern operating systems use. The program can be moved around to entirely different sections of RAM while it is actively running.
• The computer waits until the very microsecond an instruction is executed to figure out its physical address. To do this quickly without slowing down your computer, it needs special hardware assistance—specifically base and limit registers (hardware maps) that calculate the real address on the fly.
Logical vs. Virtual Space
• An operating system completely separates the addresses a program thinks it is using from the actual hardware locations in your RAM chips.
• Logical Address (Virtual Address): This is the address generated by the CPU while a program is running. The program itself, the programmer, and the CPU only ever see and deal with these logical addresses.
• Physical Address: This is the actual, real slot inside the hardware RAM memory unit where the data is stored.
Memory Management Unit
• The Memory-Management Unit, or MMU, is a dedicated hardware device inside your computer that acts as a real-time translator between the CPU and your physical RAM chips. Its primary job is to take the virtual addresses generated by a running program and map them instantly to actual physical addresses in the hardware memory.
• MMU uses a method called a relocation register scheme. When a program requests a piece of data at a specific logical address, the MMU automatically takes that number and adds it to the base value currently stored inside its relocation register.
Physical Address = Logical Address + Relocation Register
Static and Dynamic Linking
Static Linking:
Static linking happens before your program runs (at compile time). The system tool, called a linker, takes your compiled object code, finds all the required library routines on the storage disk, and physically stitches a complete copy of those libraries directly into your final executable file (like an .exe or a.out).
Dynamic Linking (The Shared Approach):
Dynamic linking solves this wastefulness by postponing the linking process entirely until execution time (when the program is actively running). Instead of packing the real library code inside the executable file, the compiler leaves a tiny placeholder called a stub.
Overlays An overlay structure allows a program to be divided into independent pieces. The system keeps in memory only the specific instructions and data needed at any given moment.

When a new piece of the program is required, it is loaded directly into the exact same physical memory space that was previously occupied by a piece no longer in use. In essence, different parts of the program "take turns" occupying the same block of RAM.
Swapping
When you have too many applications open at the same time, your physical RAM runs out of space. To prevent the system from crashing, the operating system will temporarily kick an idle or inactive process completely out of the main memory (RAM) and write its data onto your hard drive.
Later, when you click back onto that application to use it again, the OS brings it back from the hard drive and puts it back into the RAM so it can continue executing right where it left off.
• Swap-Out: Moving a process from RAM to the hard drive.
• Swap-In: Moving a process from the hard drive back into RAM.
Roll Out, Roll In
This is a variant of swapping tailored for priority-based scheduling algorithms. If a high-priority process suddenly becomes ready to run but physical memory is full, the operating system will forcefully swap out a lower-priority process (roll out) to make immediate room. Once the high-priority task completes, the original process is brought back (roll in).
3 Ways to Optimize Swapping
• Track Exact Memory: Swap out only the RAM space a program is actively using, not its entire maximum limit.
• Faster Hardware: Upgrade the backing store from slow HDDs to high-speed NVMe SSDs to boost transfer rates.
• Overlap Operations: Execute a different process in RAM while the current process waits for its data to swap to disk.
Swapping Constraints and Pending I/O
• The OS must follow strict safety rules before swapping out a process:
• The Core Constraint: A process must be completely idle before it can be safely removed from RAM.
• The Pending I/O Problem: Never swap a process waiting for input/output data (like a network download). If swapped, incoming hardware data will overwrite the new program occupying that RAM slot, corrupting memory.
• OS Buffering Solution: To bypass this, the OS routes I/O data into secure system buffers first. Once the transfer completes, the OS copies it to the program's memory, allowing the program to safely swap while the hardware transfer happens.
Bare Machines
• Advantages: Maximum user flexibility, total simplicity, and minimum hardware/software cost.
• Disadvantages: Provides zero system services. The OS has no control over interrupts, errors, system calls, or organizing job sequencing.
Memory Allocation Main memory is typically split into two distinct areas, with low memory reserved exclusively for the resident operating system kernel and high memory designated for executing user programs and processes. To map out and manage this user space, systems rely on two core approaches: single-partition allocation, which accommodates only one program at a time, or multiple-partition allocation, which enables multi-tasking.
1. The Core Protection Mechanism
To protect memory, the hardware uses two special boundary registers to frame the allowed memory space for a running process:
• Base Register: Stores the smallest (starting) physical memory address allocated to the program.
• Limit Register: Stores the total size (range) of the program's memory segment.
Together, they establish a legal range. For example, if the Base register is 300040 and the Limit register is 120900, the program can only legally access addresses between 300040 and 420940 (Base + Limit).
2. How the Hardware Checks Every Address
Every time a running program tries to read or write to a memory address, the CPU hardware automatically forces a two-step check before allowing it to pass to RAM:
1. The First Check (≥Base): The CPU verifies if the target address is greater than or equal to the value inside the Base Register.
2. The Second Check (<Base+Limit): It then checks if the address is strictly less than the combined value of the Base + Limit registers.
• If the address passes both checks: The hardware forwards the request directly to the physical RAM chip.
• If it fails either check: The hardware instantly intercepts the request, blocks the access, and traps an error to the operating system kernel, resulting in a fatal addressing error (commonly known as a segmentation fault).
3. Modifying the Boundaries
To keep the system secure, user programs are strictly forbidden from modifying these boundaries. The instructions used to alter the base and limit registers are designated as privileged instructions, meaning they can only be executed by the operating system running in kernel mode.
Skipping In a multiple-partition or fixed-partition memory allocation system, if the specific large partition a high-priority process needs is occupied, the operating system's scheduler will skip past it. It does this to load smaller, lower-priority jobs that can actually fit into currently vacant, smaller memory slots, preventing the CPU from sitting completely idle while waiting for massive blocks of RAM to clear up.
Multiprogramming with a Fixed Tasks (MFT)
1. Fixed Memory Partitioning
The system divides the available user memory into a permanent, static number of regions called partitions. The size of each partition is decided when the system is initialized and cannot be changed on the fly. While the partitions can either be all the same size (equal-sized) or different sizes (unequal-sized), the total number of partitions remains fixed.
2. Task Allocation
Each partition can hold exactly one task (or process) at a time. Because the number of partitions is locked, the maximum number of tasks the system can run simultaneously (the degree of multiprogramming) is directly limited by the number of partitions. For example, if there are 4 partitions, only 4 jobs can reside in memory at once. If a partition is empty, a waiting job from the input queue is loaded into it.
3. The Downside: Internal Fragmentation
The biggest drawback of MFT is internal fragmentation, which causes wasted memory. If a small task (e.g., 20 KB) is allocated to a large partition (e.g., 100 KB), it will occupy the entire partition. The remaining unused space (80 KB) cannot be used by any other process and goes completely to waste until that specific task finishes executing.
2 Types of MFT
• Absolute Translation and Loading: assigns a separate job queue to each individual memory region, meaning each queue is scheduled independently.
• Relocatable Translation and Loading: places all incoming jobs into a single, shared input queue, allocating memory using specific system policies whenever a partition becomes available.
Memory Allocation Policies
• First Fit: allocates the very first available hole in memory it finds that is large enough to hold the incoming job.
• Best Fit: searches through all available spaces to find and allocate the hole that is nearest to the exact size of the job.
• Worst Fit: scans memory and allocates the largest available hole or biggest region that can accommodate the job, leaving behind the largest remaining fragment.
Multiprogramming with a Variable Number of Tasks (MVT)
1. Dynamic Partitioning
When a process arrives, the operating system checks if there is enough free memory to accommodate it. If there is, the OS carves out a partition tailored exactly to the size of that specific task. Because these partitions are created on the fly, their sizes and positions change constantly as programs enter and leave the system.
2. Flexible Task Capacity
Unlike MFT, there is no hard limit on the number of tasks that can run at the same time. The degree of multiprogramming depends entirely on the size of the incoming processes and the total available memory. If you have many small programs, the system can pack a large number of them into memory simultaneously.
3. The Downside: External Fragmentation
While MVT completely eliminates internal fragmentation (since no process gets more space than it needs), it introduces external fragmentation. As processes finish and leave memory, they leave behind scattered, non-contiguous "holes" of free space. Over time, you might have plenty of total free memory to run a new program, but because that memory is split into small pieces across the system rather than one continuous block, the new program cannot be loaded.
To fix this, operating systems have to use compaction (shuffling all active processes to one end of the memory to merge the free spaces), which requires significant overhead.
Memory Allocation Policies
• Best Fit: searches all available memory blocks to allocate the smallest overall region that is big enough to hold the process.
• First Fit: scans through memory from the beginning and allocates the very first region it encounters that is big enough for the job.
• Best Available Fit: selects and allocates the smallest region specifically chosen from among the currently free memory regions.
Paging
Paging is a memory management scheme that allows a process’s physical memory space to be noncontiguous, meaning a program doesn't need to be loaded into one single, unbroken block of RAM.
• The Division: The operating system splits a program's Logical Memory into fixed-sized blocks called Pages, while splitting the hardware's Physical Memory into blocks of the exact same size called Frames.
• The Page Table: When a process executes, its individual pages are loaded into any available free frames across the RAM. The system uses a centralized Page Table to keep track of exactly which page landed in which physical frame.
• Address Translation: To find data, the CPU generates a logical address split into a page number (p) and an offset (d). The system uses the page number (p) as an index to look up the matching frame number (f) in the page table, then combines it with the original offset (d) to locate the exact physical address in RAM.

Paging
The frame table contains exactly one entry for every physical frame available in your RAM. It acts like a master checklist for physical memory, indicating whether a specific frame is free or occupied, and if it is occupied, which process it is currently allocated to.
Step 1: Check the Job's Size
When a new job arrives, the job scheduler first checks how big it is. In a paging system, the program's total size is calculated and expressed in terms of pages (logical blocks of the program).
Step 2: Check Available Physical Memory
The scheduler then scans the physical memory (RAM), which is split into frames (physical blocks of the exact same size as pages). It uses the frame table discussed above to see how many slots are empty.
Step 3: Evaluate and Allocate
• If there are enough empty frames: The OS approves the allocation and claims those physical frames for the job.
• Note: These physical frames do not need to be next to each other (contiguous); they can be scattered anywhere in RAM.
Step 4: The Page Table
Finally, the pages of the program are physically loaded into the assigned frames in RAM. Because the program's pieces might now be scattered across different parts of memory, the OS updates a Page Table specifically for that job. This table maps each logical page number to its actual physical frame number so the CPU can find it later.
Fragmentation In a paging system, external fragmentation is completely eliminated because a program can be broken up and scattered across any empty physical frames in memory. However, some internal fragmentation still occurs because memory is allocated in fixed-size blocks, meaning waste only happens inside the very last allocated frame if a program's size does not perfectly fill it.
